BaaS Storage in LiveOak

  • data obviously important for applications
  • distributed off into the cloud to be consumed by multiple client
  • centrally managed
  • want something to be easy to setup, use, manage and grow

LiveOak: Current Storage Status

  • currently support MongoDB
    • Using for both generic document storage and blob storage through GridFS
  • file based resources with git support
  • future to support different databases and other storage options
    • provide proper tools for the job

‘everything is a resource’

  • datastores are exposed as a resource like everything else in LiveOak
  • over REST and messaging (STOMP)
  • json & html representations
  • uses the same services as any other resource
    • security, interceptors, business logic, etc.

Document Storage

Mongo Module

  • acts as client to the MongoDB database
  • configuration options similar to any other MongoDB client
  • can expose multiple MongoDB servers and databases

Mongo Endpoint

  • Default endpoint
    • /storage
  • An endpoint represents a single database
  • Exposes data without adding extra cruft

Mongo Endpoint

  • /storage - a database -
  • /storage/myCollection
  • /storage/myCollection/myDoc

Basic Operations

  • LiveOak Resources
    • CRUD operations
    • Field selection and expansion
    • Offset & limit
    • Sort
  • Mongo Module Specifics

Basic CRUD Demo

Expand and Fields

  • Expand: shows a referenced object
    • /storage/foo?expand=members
  • Fields: specify fields to return
    • /storage/foo/abc?fields=price
    • /storage/foo?fields=*(price,type)
    • /storage/foo?fields=*(*)

Offset & Limit, Sort

  • Offset & Limit
    • /storage/foo?offset=50&limit=10
  • Sort
    • /storage/foo?sort=price (ascending)
    • /storage/foo?sort=-price (descending)

Query

  • uses the ‘q’ query parameter
  • MongoDB Specific
    • URL encoded mongo query object
  • /storage/products?q={price:{$lt:350}}
  • /storage/products?q=%7Bprice%3A%7B%24lt%3A350%7D%7D

Aggregation

  • Aggregate query results aren’t collection items, therefore a different endpoint
    • /storage/collection/_aggregate
  • What does this aggregate query do:
    • ?q=[{$group:{_id:{country:'$country'},numPeople:{$sum:1}}}]
  • Demo using Aggregation

Referencing Documents: MongoDB

  • Manual Storage
    • manually store id in a document
    • fetch original document and request the other based on id
  • DBRef
    • special object in MongoDB
    • normally has support built in to the mongo clients
    • same fetch and request linked document lifecycle
    • DBRef("collectionName", "id")

DBRefs in LiveOak

  • Create/Update
    • fieldName: {“$dbref” : “/storage/collectionName/id”}
  • Read 
    • fieldName : {
   "id" : "[id]",
   "self" : { "href" : "/storage/collectionName/id"}
}

Advanced Features & Debugging

  • hint: specify which index to use
    • ../myCollection?sort=price&hint=price_1
  • explain: output what was used for a query
    • ./myCollection?sort=price&explain=true

Blob Storage

Mongo GridFS

  • Overloaded term - liveoak module vs. com.mongo.gridfs API
  • mongo-gridfs liveoak module adds a filesystem notion on top of mongo gridfs API

  • Blob storage with mongodb as underlying store
  • Scaling and high availability handled by mongodb infrastructure

Mongo GridFS Features

  • Every file / directory has a unique pathname, and unique oid - two ways of addressing
  • Every file / directory can have arbitrary metadata
  • Queries can be performed over all files for metadata
  • Automatic parent path creation

Mongo GridFS URL structure

  • By default bound to /gridfs
  • User view of repository: /gridfs/{userid}
  • That's where content root begins for pathname navigation
  • .files subcollection lists all the files via oid
  • .blobs subcollection allows access to blob content via oid
